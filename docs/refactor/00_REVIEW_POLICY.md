# pdf-lay リファクタリング レビュー方針 & タスク仕様テンプレート

**対象読者:** 本リファクタリングを実装するコーディングエージェント（Codex 等を含む）およびレビュアー
**位置づけ:** すべてのフェーズ設計書（`phase0_*.md` 〜 `phase4_*.md`）は本書のルールとテンプレートに従う。
本書と各設計書が食い違う場合は**本書を優先**する。

---

## 0. この文書の使い方（実装エージェント向け）

1. まず本書を全部読む。特に §2「作業ライフサイクル」、§4「Definition of Done」、§6「禁止事項」。
2. 担当フェーズの設計書を開く。設計書は**独立したタスク（`P0-1` 等）の集合**になっている。
3. **1タスク = 1ブランチ = 1PR** を厳守。複数タスクをまとめない。小さく出す。
4. タスクは**設計書に書かれた順序（依存関係）**で実施する。依存タスクが未マージなら着手しない。
5. 各タスクの「受け入れ基準」を満たし、「検証コマンド」が全て緑になってから PR を出す。
6. 判断に迷ったら**推測で実装しない**。§7「エスカレーション」に従い、質問を PR 説明に明記して人間レビューを待つ。

> **原則:** このリポジトリの既存コードのスタイル（命名・コメント密度・エラー処理の書き方）に合わせること。
> 新しい流儀を持ち込まない。周囲のコードを読んで真似る。

## 1. 全体の設計原則（すべてのフェーズに共通）

これらは各タスクが守るべき「憲法」。逸脱する変更は原則却下。

1. **No Silent Drop（テキストを黙って捨てない）**
   テキスト・行・ブロックを破棄する箇所を新設・変更する場合、必ず次のいずれか:
   (a) 最近傍の受け皿へ割り当てて破棄を回避する、または
   (b) `AnalysisResult.warnings` に破棄理由と量を記録する。
   「条件に合わないから無言で `continue`／`filter` で消す」を新たに作らない。

2. **単一の高忠実度レンダリングコア**
   出力（markdown / llm_text / chunk / json）は同一の「ブロック→リッチテキスト」変換を通す。
   ある出力にだけ数式変換やテーブル描画を足して他に足さない、という忠実度の乖離を作らない。

3. **分類は一度だけ**
   `BlockClassifier` が決めた `block_type` を下流（ヘッダー検出等）で再導出しない。分類結果を供給して使う。

4. **後方互換はフラグと `#[serde(default)]` で担保**
   デフォルト挙動を変える場合は、旧挙動に戻せる設定・フラグを残す（各設計書が明示）。
   設定構造体へのフィールド追加は必ず `#[serde(default)]` を付ける。

5. **パニックしない**
   ライブラリ経路（`pdf-lay-core`）で `unwrap()`/`expect()`/`panic!`/`unreachable!`/添字 `[i]` を
   ユーザー入力由来の値に対して使わない。`Result`/`Option` で扱う。テストコードは例外。

6. **マジックナンバーは設定へ**
   新しい閾値定数をコードに直書きしない。`Config`/各 `*Config` に追加し、既定値を持たせる。

## 2. 作業ライフサイクル（1タスクの流れ）

```
① 設計書のタスクを読む（依存タスクがマージ済みか確認）
② ブランチを切る:  git checkout -b <親ブランチ>-<task-id>   例: claude/pdf-lay-refactor-zw7i9g-p0-1
③ 実装（設計書の「実装手順」に従う。スコープ外に手を出さない）
④ セルフチェック（§3 のチェックリストを自分で通す）
⑤ 検証コマンド（§5）を全部走らせて緑を確認
⑥ コミット（§2.1 の規約）→ PR（§2.2 のテンプレート）
⑦ レビュー指摘に対応。マージ後、次のタスクへ
```

### 2.1 コミット規約

- Conventional Commits 準拠: `feat(core): ...` / `fix(extract): ...` / `test(chunker): ...` / `docs: ...`
- 1コミットは論理的に1まとまり。無関係な整形を混ぜない。
- コミット本文に**どのタスクIDか**を必ず書く（例: 末尾に `Task: P0-1`）。
- モデル識別子やセッション URL 等の内部情報を含めない。

### 2.2 PR 説明テンプレート

```markdown
## タスク
P0-1: <一行タイトル>

## 変更内容
- <何を変えたか。箇条書き>

## 受け入れ基準の充足
- [ ] <設計書の受け入れ基準1> → <どう満たしたか>
- [ ] <受け入れ基準2>

## 検証結果
- cargo fmt --check: OK
- cargo clippy -- -D warnings: OK
- cargo test -p <crate>: OK（<新規テスト名>を含む）
- <該当すれば> 実PDFでの手動確認: <結果>

## 非スコープ / 残課題
- <このPRでやらなかったこと。次タスクに送ったもの>

## レビュアーへの質問（あれば）
- <判断を仰ぎたい点>
```

## 3. セルフチェックリスト（実装者が PR 前に自分で通す）

- [ ] スコープは担当タスク1つだけか（他タスク・無関係リファクタを含めていないか）
- [ ] §1 の原則6項目に違反していないか（特に No Silent Drop / パニック禁止）
- [ ] 新しい閾値をコード直書きしていないか（`Config` へ入れたか）
- [ ] 公開API（`pub fn`/`pub struct`）を変えたなら、呼び出し側（CLI・Python・他crate）を全て更新したか
- [ ] `#[serde(default)]` を新規フィールドに付けたか
- [ ] 設計書の「追加するテスト」を全て書き、緑になっているか
- [ ] ドキュメントコメント（`///`）を追加/更新したか。**コメントと実装が食い違っていないか**（本リファクタの発端はコメントの虚偽）
- [ ] 既存テストを壊していないか。壊れたなら「仕様変更として正当か」を PR に説明したか

## 4. Definition of Done（全タスク共通の完了定義）

タスクは以下を**すべて**満たして初めて完了:

1. 設計書に列挙された**受け入れ基準（Given/When/Then）が全て満たされている**。
2. 設計書に列挙された**新規テストが全て存在し緑**、かつ既存テストが緑（正当な仕様変更を除く）。
3. `cargo fmt --check` と `cargo clippy --all-targets -- -D warnings` が緑。
4. 公開APIの変更が**呼び出し側3面（CLI / Python / `pdf-lay` crate）に反映**されている。
5. ドキュメントコメントが実装と一致している。
6. PR 説明テンプレート（§2.2）が埋まっている。

## 5. 検証コマンド（標準セット）

```bash
# 整形
cargo fmt --all --check

# 静的解析（警告をエラー扱い）
cargo clippy --all-targets --all-features -- -D warnings

# 単体・結合テスト（crate単位で絞ると速い）
cargo test -p pdf-lay-core
cargo test -p pdf-lay-cli
cargo test --workspace

# Python バインディング（該当タスクのみ）
uvx maturin develop -m crates/pdflay-python/Cargo.toml
python -c "import pdflay; print(pdflay.analyze.__doc__)"

# CLI 手動スモーク（実PDFフィクスチャ導入後）
cargo run -p pdf-lay-cli -- toc tests/fixtures/<sample>.pdf
cargo run -p pdf-lay-cli -- markdown tests/fixtures/<sample>.pdf -o /tmp/out.md
```

各設計書は、そのタスク固有の追加検証（実PDFでの網羅率確認など）を「検証」節で指定する。

## 6. 禁止事項（Hard No — 見つけたらレビューで即差し戻し）

- **スコープクリープ:** 担当タスク以外のコードを「ついでに」直す。整形だけの巨大diff。
- **Silent Drop の新設:** 条件に合わない要素を warning なしで捨てる新コード。
- **ライブラリ経路での panic:** `unwrap`/`expect`/添字アクセスをユーザー入力由来値に使う。
- **コメントと実装の乖離:** 「MediaBoxをパース」と書いて定数を返す類（本リファクタの元凶）。
- **テスト削除でごまかす:** 落ちたテストを「不要」と称して消す（正当性を PR で説明できない限り禁止）。
- **設定の直書き:** 新しい閾値を `Config` を経由せずコードに埋める。
- **公開APIの片手落ち変更:** シグネチャを変えて CLI か Python の呼び出し側を放置しビルドを壊す。
- **推測実装:** 仕様が曖昧なまま「たぶんこう」で書く（→ §7 へ）。

## 7. エスカレーション（迷ったときの手順）

判断が設計書だけで決まらない場合:

1. **推測で実装しない。**
2. 最小限の安全側（No Silent Drop・後方互換維持・パニック回避）に倒した上で、
3. 迷った点を **PR 説明の「レビュアーへの質問」に具体的に書く**（選択肢A/Bと自分の暫定案を提示）。
4. 該当タスクの受け入れ基準に影響するなら、その基準を満たせない旨を明記し、人間の指示を待つ。

「わからないことを黙って埋める」より「明示して止まる」を評価する。

## 8. テスト方針

### 8.1 テストの3層

| 層 | 目的 | 置き場所 |
|----|------|----------|
| ユニット | 関数単位の分岐・境界値。合成データで決定的に | 各モジュール `#[cfg(test)] mod tests` |
| ゴールデン | 実PDF→出力の回帰。網羅率/読み順/セクション精度 | `tests/fixtures/*.pdf` + 期待値 |
| スモーク | CLI/Python が落ちずに動く | `tests/integration/`, `crates/pdf-lay/tests/` |

### 8.2 ゴールデン用フィクスチャ（横断タスク X-1 で整備）

現状 `tests/fixtures/sample.pdf` が無く主要結合テストが `#[ignore]`（`pipeline.rs:333`）。
最低限、以下の実PDFを用意し、各フェーズの回帰基準にする:

- A4・1段組・**日本語**論文（①ページ寸法/CJK の検証に必須）
- IEEE 2段組（ローマ数字見出し・走りヘッダ）
- arXiv preprint（LaTeX由来数式・多様な見出し）
- スキャン/画像onlyページを含むPDF（抽出限界の検証）

### 8.3 品質メトリクス（CIで継続測定、仕様書 §3.2 の目標に対応）

- テキスト網羅率 = 出力到達文字数 / 抽出スパン総文字数（目標: 高いほど良い、回帰させない）
- セクション検出（見出しの適合率/再現率、目標 >95%）
- キャプション-画像マッチ率（目標 >90%）
- 読み順正確性（人手アノテーションとの一致、目標 >95%）

各タスクは「この変更で上記メトリクスを悪化させない」ことを回帰テストで担保する。
